# 明日方舟：终末地 限定角色池抽卡计算器

一个零后端、纯静态的抽卡概率 & 期望计算器。核心结果由**精确动态规划（DP）**计算得出，**不是 Monte Carlo 模拟**。当前以限定角色"庄方宜"为示例，机制写成可复用结构，后续替换限定 up 时只需改一处配置。

在线体验：部署到任意 GitHub Pages 即可。

---

## 功能

- **概率计算**：输入嵌晶玉、限定抽、已垫抽数、目标限定 up 数量，输出
  - 成功率（至少获得目标数量的概率）
  - 期望获得数量、至少 1 个的概率
  - 完整概率分布表（严格从 `0` 到理论可达最大值，不合并尾桶）
- **期望计算**：输入目标限定 up 数量，输出
  - 从全新当前池状态开始所需的**期望主抽数**
  - 严格有限上界提示（`K=1` 时 120 主抽；`K≥2` 时 `240×(K-1)` 主抽必达成）

## 规则摘要（与页面一致）

- 1 限定抽 = **500 嵌晶玉**（不足 500 的余玉不折算）
- 6★ 基础概率 **0.8%**，pity 65 起每抽 +5pp，第 80 抽硬保底；任意 6★ 清零 pity
- 6★ 中**限定 up 概率 50%**（本体或信物都算 1 个限定 up；信物规则使代码无需区分两者）
- 当前池**前 120 主抽内必得 1 次限定 up**（每池仅生效一次）
- 当前池第 **30 主抽**赠送 **1 次免费十连**（每池仅一次）
- 当前池**每 240 主抽**赠送 **1 个限定信物**（240/480/720…）
- 免费十连**不计入**：主抽 pity / 240 奖励累计 / 120 主抽计数
- 免费十连**若出限定 up**：会消耗/满足 120 保底资格（`hasLimited` 置 true）

已忽略（按用户要求）：60 抽奖励。

## 已知假设与输入约定

- 当前限定池默认为**全新状态**：累计 0 抽、120 保底未触发、免费十连未领、240 奖励累计 0。
- 唯一允许的"中途"输入是**已垫抽数**（0–79），仅用于普通主抽的 6★ pity；不代表当前池已抽过。
- 概率计算假设用户**把所有可用主抽花完**（不是"达到目标就停"）。
- 期望计算假设"达到目标就停止"，**只统计普通主抽消耗**（不含免费十连内部的 10 次出手）。
- 期望计算始终从 fresh 状态开始，不读取"已垫抽数"（这是"期望总消耗"的定义）。

## 算法说明

### 状态表示

```
state = (pityCount ∈ [0, 79], hasLimited ∈ {0, 1}, limitedCount ∈ [0, maxReachable])
```

其中 `maxReachable = N + (N ≥ 30 ? 10 : 0) + ⌊N/240⌋`，是理论可达最大值。

### 单次普通主抽转移

设第 `nextDraw = mainDrawsDone + 1` 次主抽：

- 若 `hasLimited == false && nextDraw == 120`：强制限定 up（概率 1，lim+1，pity=0，hasLimited=true）
- 否则按 `p6 = p6ForPity(pity)`：
  - `p6 × 0.5` → 限定 6★（lim+1，pity=0，hasLimited=true）
  - `p6 × 0.5` → 非限定 6★（pity=0）
  - `1 - p6` → 未出 6★（pity+1）

其中

```
p6ForPity(pity) = 0.008                              ; pity ≤ 64
                  0.008 + 0.05 × (pity - 64)         ; 65 ≤ pity ≤ 78
                  1.0                                ; pity == 79
```

### 免费十连

插入于第 30 次主抽结算之后。关键推导：

> 第 10 抽的两种情况（前 9 抽全 4★ vs 至少一个 5+）下，**6★ 边际概率都是 0.8%**。  
> 5+ 保底只改变了 5★/4★ 占比，并不影响 6★ 分布。

因此免费十连：

- 10 抽内 6★ 数量 ~ `Binomial(10, 0.008)`
- 10 抽内限定 up 数量 ~ `Binomial(10, 0.004)`（每个 6★ 独立 50% 为限定 up）

免费十连只修改 `limitedCount` 和 `hasLimited`，不改 `pityCount` 和 `mainDrawsDone`。

### 240 抽送信物

每次主抽结算完成后，若 `mainDrawsDone % 240 == 0`，则 `limitedCount += 1`。

### 概率计算

从 fresh 状态 + 用户的 `pityStart` 出发，对每次主抽按上述转移传播概率分布。最终对所有终态按 `limitedCount` 聚合，得到完整分布 `dist[0..maxReachable]`。

### 期望计算（吸收态 DP）

- 从 fresh 状态出发（`pityStart = 0`）；
- 严格上界 `N_MAX = (K==1) ? 120 : 240×(K-1)`：
  - `K=1`：120 主抽必触发限定保底 → 一定达成；
  - `K≥2`：每 240 主抽送 1 个信物，外加第 120 主抽保底，因此 `240×(K-1)` 必达成；
- 每步把 `limitedCount ≥ K` 的概率质量作为**吸收态**按当前 `mainDrawsDone` 累加到期望中；
- `E = Σ k × P(首次在第 k 次主抽达成目标)`。

### 为什么不是 Monte Carlo？

精确 DP：

- 分布与期望**数学精确**（浮点误差 < 1e-14）；
- 每次计算**结果可复现**；
- 长概率尾端的微小差异（例如 `P(lim = 15) < 0.0001%`）依然可靠。

Monte Carlo 要达到同样的小概率精度需要几亿次样本，且每次结果抖动。这里状态空间极小（约 80×2×maxReachable），DP 明显是更好的选择。

## 如何替换成别的限定角色

- 打开 `script.js` 顶部的 `CONFIG` 对象，修改 `characterName` 字段为新角色名。
- 打开 `index.html`，修改顶部 `<h1>` 标题与规则摘要文案（结构可完全复用）。
- 机制规则（120 保底 / 30 免费十连 / 240 信物 / 6★ pity）不变时，无需改算法。
- 若后续新限定角色的基础概率、软保底、硬保底有调整，只需修改 `CONFIG` 中的
  `baseP6` / `pitySoftStart` / `pityHardCap` / `pityStep` / `limitedRate`
  等字段。

## 自测用例（页面打开时自动运行，控制台查看）

1. `N=0, target=1` → 成功率 0%
2. `N=120, pityStart=0, target=1` → 成功率 100%
3. `N=240, target=2` → 成功率 100%
4. `N=29` 不触发免费十连；`N=30` 触发 1 次
5. `pityStart=79, N=1` → `dist = [0.5, 0.5]`（首抽必出 6★，50% 限定）
6. `E[K=1].bound == 120`，`E[K=2].bound == 240`
7. 多个 N 下概率分布和 ≈ 1
